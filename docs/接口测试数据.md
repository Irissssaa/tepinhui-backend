## 六、商家入驻模块 `/tph/api/v1/merchant`

### 6.1 商家入驻申请（用户）

**业务逻辑：**
```
1. 从 JWT 解析当前用户（必须是 CONSUMER 角色）
2. 检查该用户是否已有 pending/approved 商家记录
   → 有 → 抛 400 "您已有入驻申请"
3. 检查用户角色是否为 ADMIN/MERCHANT
   → 是 → 抛 403 拒绝
4. 插入 merchant 表，status = "pending"
5. 返回新商家 ID
```

**注：** rejected 状态的用户可以重新申请（被拒绝后修改资料再次提交）。

```
POST /api/v1/merchant/apply
Header: Authorization: Bearer <consumer用户token>
Content-Type: application/json

{
    "shopName": "赣南脐橙旗舰店",
    "licenseNo": "91360000MA35XXXXX",
    "qualification": "https://oss.example.com/license/gannan.jpg"
}
```

**响应示例：**
```json
{
    "code": 200,
    "message": "商家入驻申请已提交",
    "data": 1
}
```

**常见错误响应：**
```json
// 角色不符（ADMIN/MERCHANT 用户申请）
{"code": 403, "message": "您已是商家，无需重复申请"}

// 重复申请
{"code": 400, "message": "您已有入驻申请，无需重复提交"}

// 未登录
{"code": 401, "message": "未登录"}
```

### 7.4 审核商家（管理员）

**业务逻辑：**
```
1. 检查 ADMIN 角色
2. 查询该商家记录，确认 status = "pending"
   → 不是 pending → 抛 400 "该商家不是待审核状态"
3. 更新 merchant.status + auditRemark
4. 如果 status = "approved"：
   a. 查出对应 user 记录
   b. user.role = MERCHANT
   c. 更新 user 表
   d. 该用户需重新登录获取含 MERCHANT 角色的新 token
```

**通过审核：**
```
PUT /api/v1/admin/merchant/{id}/audit
Header: Authorization: Bearer <admin用户token>
Content-Type: application/json

{
    "status": "approved",
    "auditRemark": "资质齐全，审核通过"
}
```

**驳回申请：**
```
PUT /api/v1/admin/merchant/{id}/audit
Header: Authorization: Bearer <admin用户token>
Content-Type: application/json

{
    "status": "rejected",
    "auditRemark": "营业执照信息与注册信息不符，请重新上传"
}
```

**status 可选值：** `approved`（通过）、`rejected`（驳回）

## 八、平台看板 `/tph/api/v1/admin/stats`

### 8.1 平台数据看板（管理员）

**业务逻辑：**
```
1. 检查 ADMIN 角色（SecurityConfig 自动拦截）
   → 无 ADMIN 角色 → 403 Forbidden
2. 查 Redis 缓存 key="admin:stats"
   → 有缓存 → 直接返回
3. 无缓存 → 查 5 张表统计数字：
   - user 表总记录数 → userCount
   - merchant 表 status=approved → merchantCount
   - merchant 表 status=pending → pendingMerchantCount
   - product 表总记录数 → productCount
   - orders 表总记录数 → orderCount
   - orders 表 status in (done, shipped) 的 total_amount 求和 → totalSalesAmount
   - trace_record 表总记录数 → traceCount
   - trace_record 表 audit_status=pending → pendingTraceCount
4. 写入 Redis，TTL=30分钟
5. 返回 AdminStatsVO
```

**缓存说明：** 首次访问查数据库，后续30分钟内直接读缓存。需刷新数据时等待TTL过期或手动清缓存。

```
GET /api/v1/admin/stats
Header: Authorization: Bearer <admin用户token>
```

**响应示例：**
```json
{
    "code": 200,
    "data": {
        "userCount": 25,
        "merchantCount": 3,
        "pendingMerchantCount": 1,
        "productCount": 12,
        "orderCount": 8,
        "totalSalesAmount": 2846.00,
        "traceCount": 5,
        "pendingTraceCount": 1
    }
}
```

**字段说明：**

|| 字段 | 数据来源 | 说明 |
||---|---|---|
| userCount | user 表 COUNT(*) | 全部注册用户 |
| merchantCount | merchant WHERE status='approved' | 审核通过的商家 |
| pendingMerchantCount | merchant WHERE status='pending' | 待审核商家 |
| productCount | product 表 COUNT(*) | 全部商品 |
| orderCount | orders 表 COUNT(*) | 全部订单 |
| totalSalesAmount | orders WHERE status IN (done,shipped) SUM(total_amount) | 已完成+已发货订单的销售总额 |
| traceCount | trace_record 表 COUNT(*) | 全部溯源记录 |
| pendingTraceCount | trace_record WHERE audit_status='pending' | 待审核溯源 |

**常见错误响应：**
```json
// 未登录
{"code": 401, "message": "未登录"}

// 非管理员
{"code": 403, "message": "禁止访问"}
```

## 第九章 单元测试（AdminStatsServiceTest）

`src/test/java/com/tepinhui/tepinhui_backend/service/AdminStatsServiceTest.java`

### 测试文件说明

4个测试用例，验证 `AdminStatsServiceImpl.getAdminStats()` 的缓存命中、数据库查询、边界条件处理：

| 测试用例 | 验证点 |
|---------|--------|
| `getAdminStatsShouldReturnCachedResultWhenAvailable` | 缓存命中时直接返回，不查库 |
| `getAdminStatsShouldQueryAllTablesAndCacheWhenCacheMiss` | 缓存未命中时查5张表，验证各字段值正确，缓存写入30min |
| `getAdminStatsShouldHandleNullTotalAmount` | 订单金额为null时 reduce 过滤后仍正确累加 |
| `getAdminStatsShouldReturnZeroWhenNoOrders` | 无订单时 totalSalesAmount 返回 BigDecimal.ZERO |

### 运行方式

```bash
./mvnw test -Dtest=AdminStatsServiceTest
```

### 已知遗留问题

- `ProductControllerTest.createProductShouldRejectConsumerRole` 原有失败（与本次修改无关）：匿名用户 POST /api/v1/products 期望403但返回200，原因是 SecurityConfig 对该端点的角色限制配置或测试上下文与实际行为不一致。
