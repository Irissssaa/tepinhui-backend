0. 当前仓库状态同步（T12）

本节用于同步当前代码仓库的真实完成边界，优先级高于下文的原始“待开发”规划描述。当前代码已经补齐大部分模块的 Controller、DTO/VO、Swagger 注解、安全规则和框架级测试，但除少数模块外，很多接口仍处于“契约已定、业务待补”的阶段。

0.1 状态口径

  • 已实现业务逻辑：已有真实 Service 处理流程，至少覆盖主要数据读写、权限控制、缓存或状态流转中的核心部分。
  • 接口框架已搭建，业务未实现：接口路径、请求/响应契约、Swagger 文档、安全规则和基础测试已就位，但 Service 仍返回空结构占位数据，或抛出 501 类业务待实现异常。
  • 部分实现：接口已可调用，且实现了部分核心逻辑，但仍存在硬编码、联查缺失、示例数据或链路未闭环的问题。

0.2 Swagger （未实现） 标记说明

Swagger 中标注 （未实现），表示该接口的路径、鉴权规则、请求参数、响应结构和统一返回格式 Result<T> 已确定，可用于前后端联调，但当前 Service 没有真实业务实现。

0.4 当前角色约束

  • 普通注册固定创建为 CONSUMER。
  • 商家角色 MERCHANT 不通过普通注册直接授予，应在商家入驻审核通过后由后端执行角色流转。
  • 管理员角色 ADMIN 不开放普通注册，只能由受控后台或初始化脚本创建。

0.5 后续业务实现优先级

  1. 先完成商家审核与用户角色流转，打通 CONSUMER -> MERCHANT 的真实授予链路。
  2. 再完成购物车与订单，补上下单、库存、支付前后状态和评价闭环。
  3. 最后补平台看板和文化内容。

三、溯源码规范（严格按 trace_solution.html）

3.1 溯源码命名规则

TP - {省份缩写} - {品类缩写} - {年份} - {6位随机数}

示例： TP-ZJ-LJ-2024-042138

字段 含义 说明

2024 生产年份 四位年份
042138 随机批次号 6位数字，保证唯一性

3.2 省份/品类缩写字典

省份： 北京→BJ、天津→TJ、河北→HB、浙江→ZJ、江苏→JS、上海→SH 等34个省市

品类： 龙井茶→LJ、碧螺春→BLC、铁观音→TG、普洱茶→PE、茶叶→CY 等

四、溯源查询流程（严格按 trace_solution.html）

4.1 查询链路（6步）

1. 消费者触发查询（扫描二维码 / 手动输入溯源码）
        ↓
2. 后端优先查 Redis 缓存（key: trace:{traceCode}）
        ↓（未命中）
3. 查询 MySQL：trace_record WHERE trace_code = ? AND audit_status = 'pass'
        ↓
4. 联查 product、merchant 表，组装完整数据
        ↓
5. 写入 Redis 缓存（TTL=24h），返回前端
        ↓
6. 前端按六层结构渲染：产品 → 产地 → 生产 → 加工 → 质检 → 流通

4.2 响应结构（6层嵌套）

层级 字段
traceCode 溯源码
product name, spec, merchantName, coverImg
origin province, city, county, address, longitude, latitude
production produceDate, producer, rawMaterial
process factory, processDate, processDesc
inspection org, inspectDate, result, reportUrl
logistics warehouseIn, warehouseOut, logisticsInfo
