## 三、溯源码规范（严格按 trace_solution.html）

### 3.1 溯源码命名规则

```
TP - {省份缩写} - {品类缩写} - {年份} - {6位随机数}
```

**示例：** `TP-ZJ-LJ-2024-042138`

| 字段 | 含义 | 说明 |
|---|---|---|
| `TP` | 前缀 | 固定为 TP |
| `ZJ` | 省份缩写 | 浙江 |
| `LJ` | 品类缩写 | 龙井 |
| `2024` | 生产年份 | 四位年份 |
| `042138` | 随机批次号 | 6位数字，保证唯一性 |

### 3.2 省份/品类缩写字典

**省份：** 北京→BJ、天津→TJ、河北→HB、浙江→ZJ、江苏→JS、上海→SH 等34个省市

**品类：** 龙井茶→LJ、碧螺春→BLC、铁观音→TG、普洱茶→PE、茶叶→CY 等

## 四、溯源查询流程（严格按 trace_solution.html）

### 4.1 查询链路（6步）

```
1. 消费者触发查询（扫描二维码 / 手动输入溯源码）
        ↓
2. 后端优先查 Redis 缓存（key: trace:{traceCode}）
        ↓（未命中）
3. 查询 MySQL：trace_record WHERE trace_code = ? AND audit_status = 'pass'
        ↓
4. 联查 product、merchant 表，组装完整数据
        ↓
5. 写入 Redis 缓存（TTL=24h），返回前端
        ↓
6. 前端按六层结构渲染：产品 → 产地 → 生产 → 加工 → 质检 → 流通
```

### 4.2 响应结构（6层嵌套）

| 层级 | 字段 |
|---|---|
| traceCode | 溯源码 |
| product | name, spec, merchantName, coverImg |
| origin | province, city, county, address, longitude, latitude |
| production | produceDate, producer, rawMaterial |
| process | factory, processDate, processDesc |
| inspection | org, inspectDate, result, reportUrl |
| logistics | warehouseIn, warehouseOut, logisticsInfo |

### 4.3 缓存策略

| 缓存Key | TTL | 接口 |
|---|---|---|
| `trace:{traceCode}` | 24小时 | 溯源查询 |
| `trace:stats` | 1小时 | 溯源统计 |
| `stats:specialty:map` | 2小时 | 全国特产分布地图 |
| `stats:sales:region:{region}` | 1小时 | 地区销售数据 |
| `stats:trace:region:{region}` | 1小时 | 地区溯源数据 |
| `product:list:{page}:...filters` | 5分钟 | 商品列表（实际实现按分页和筛选条件区分缓存键） |
| `product:detail:{id}` | 10分钟 | 商品详情 |
